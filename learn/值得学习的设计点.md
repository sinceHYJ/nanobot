# nanobot AgentLoop 值得学习的设计点

> 从 `nanobot/agent/loop.py` 与 `nanobot/agent/runner.py` 中提炼的工程设计模式。
> 每一处都附源码位置，便于回溯。

---

## 一、状态机管线：把巨型函数拆成可观测的阶段

**源码**：`loop.py:100-246`（`TurnState` + `_TRANSITIONS`）、`loop.py:1418`（分发循环）

### 要点

- **约定式反射分发**：`_state_{state.name.lower()}` 命名约定让 handler 自动绑定，加阶段只需加枚举值 + 方法 + 转移规则，零注册。
- **声明式转移表**：`(state, event) → next_state` 集中管理，一眼看清全部合法路径（含 shortcut 短路），比散落在 if/else 里的隐式控制流可维护得多。
- **每阶段独立可测**：每个 `_state_*` 只做一件事，输入 `TurnContext` → 修改它 → 返回事件字符串，边界清晰，可以单独写单测。

### 适用场景

任何"请求处理管线"——消息队列消费、HTTP 中间件、CI/CD 流水线、ETL。

## 二、Trace 内建可观测性

**源码**：`loop.py:113-119`（`StateTraceEntry`）、`loop.py:1429-1453`（记录每阶段耗时+事件）

### 要点

不是"事后加日志"，而是**在状态机循环里自动记录每个阶段的 `started_at / duration_ms / event / error`**，形成完整执行轨迹。出了问题可以直接拿 `ctx.trace` 排查，不需要重新复现。

这比"在关键函数前后手动 `logger.info`"更可靠——**框架保证不遗漏**。

### 适用场景

任何需要性能分析或故障排查的长流程系统。

---

## 三、上下文治理与持久化解耦（append-only 边界）

**源码**：`runner.py:352-365`（`inflight_start_index` + 注释）、`_run_core` 中 `messages_for_model` 与 `messages` 分离

### 要点

这是整段代码里**最精妙**的设计之一。同一个 `messages` 列表有两个角色：

- **给模型看的副本**（`messages_for_model`）：可以被治理器修复、压缩、剥离占位消息
- **写回 session 的原始列表**（`messages`）：保持 append-only，治理产生的"合成编辑"绝不污染它

注释点明了原因：*"synthetic edits must not shift the append boundary used later when the caller saves only the new turn"*。

这个模式的价值：**读路径（给模型）和写路径（持久化）的关注点完全隔离**，类似数据库的"读写分离"。

### 适用场景

任何需要"给下游的消费版本"和"持久化的权威版本"不一致的场景（如日志美化 vs 原始日志、缓存 vs 源数据）。

## 四、四阶段 checkpoint 崩溃恢复

**源码**：`loop.py:233`（`_RUNTIME_CHECKPOINT_KEY`）、`runner.py` 四次 `_emit_checkpoint`、`loop.py:1889`（`_restore_runtime_checkpoint`）

### 要点

把一次回合的执行切成 `awaiting_tools → tools_completed → final_response` 三个检查点写入 session metadata。崩溃后 `_restore_runtime_checkpoint` 会：

- 重放已完成的 assistant_message + tool_results
- 对未完成的 pending_tool_calls 生成一条 `"Error: Task interrupted"` 的占位 tool 结果

这样 LLM 下次请求时能看到完整的对话结构（每个 tool_call 都有对应 tool_result），不会因为"孤儿 tool_call"导致 provider 报错。

### 适用场景

任何长时间运行、可能中途崩溃的异步任务系统——尤其是"外部状态机"恢复（对比 Airflow 的 task instance 状态）。

## 五、工具并发 + 依赖分批

**源码**：`runner.py:1083-1130`（`_execute_tools` + `_partition_tool_batches`）

### 要点

不是简单地"所有 tool_call 并发"或"全串行"，而是：

1. 先按依赖/冲突关系分批（`_partition_tool_batches`）
2. 同一批内无依赖 → `asyncio.gather` 并发
3. 批次之间串行等待

这比"无脑并发"安全（避免写写冲突），比"全串行"高效。类似数据库事务的 batch commit 思路。

### 适用场景

任何有依赖关系的并发任务调度（工作流引擎、并发 IO 调度）。

---

## 六、注入队列：中途消息的有序消费

**源码**：`loop.py:802-860`（`_drain_pending`）、`runner.py:149-210`（`_try_drain_injections`）

### 要点

一个回合（turn）执行期间，用户可能继续发消息、子 agent 可能返回结果。这套设计用 `pending_queue` + **三个注入检查点**（工具执行后 / 最终响应前 / 错误后）保证：

- 后续消息**按序注入**到当前对话，而不是触发独立的新回合
- 有 `injection_cycles` 上限防止无限循环
- `_drain_pending` 在"队列空但子 agent 还在跑"时会阻塞等待（最多 300s），避免结果丢失

### 适用场景

任何"主任务执行中需要消费异步到达的副作用"的系统（actor model、event sourcing）。

---

## 七、容错降级链

**源码**：`runner.py:367-386`（治理失败降级）、`_run_core` 中空响应重试 + length 恢复 + finalization retry

### 要点

每个可能失败的点都有**逐级降级**策略，而不是直接抛异常：

- 上下文治理失败 → 降级到最小修复（strip + backfill）
- LLM 返回空 → 重试 N 次 → 再做无工具 finalization 请求
- `finish_reason=length` → 追加恢复消息继续生成
- max_iterations 耗尽 → 可选无工具收尾请求 → 兜底消息

**设计原则**：*"永远给用户一个可用的结果，哪怕质量降级"*。对比"一处报错整个请求失败"的脆弱设计。

### 适用场景

任何面向用户、需要高可用性的系统（推荐系统降级、CDN 回源、微服务熔断）。

---

## 八、Hook 钩子链保证完整性

**源码**：`runner.py:274-310`（`AgentRunner.run` 的 try/except/finally）

### 要点

`before_run → _run_core → after_run/on_error → on_finally` 的钩子链设计：

- `finally` 里即使前面异常也会执行 `on_finally`
- `on_finally` 自身报错不会吞掉原始异常（用 try/except 包住 + `logger.exception`）
- 每个迭代步骤都有 `before_iteration` / `after_iteration` 细粒度钩子

这让**流式输出、进度回调、指标采集**等横切关注点可以挂载到任意阶段，而不侵入核心逻辑。

### 适用场景

任何需要插件化扩展的框架（Web 框架中间件、任务调度器钩子）。

---

## 总结：值得带走的设计原则

| 原则 | 对应代码 |
|---|---|
| 把巨型流程拆成状态机 | `TurnState` + `_TRANSITIONS` |
| 框架内建可观测，不靠事后加日志 | `StateTraceEntry` |
| 读路径与写路径隔离 | `messages_for_model` vs `messages` |
| 检查点支撑崩溃恢复 | 四阶段 checkpoint |
| 并发按依赖分批，不做全/无选择 | `_partition_tool_batches` |
| 异步副作用用队列+检查点有序消费 | `_drain_pending` + `_try_drain_injections` |
| 每个失败点都有降级策略 | 容错降级链 |
| 钩子链管理横切关注点 | `AgentRunner.run` |

这些设计点共同构成了一个**"可在生产环境可靠运行的 agent 执行引擎"**的核心骨架，值得在自己的系统设计里复用。
